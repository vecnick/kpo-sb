# 🧾 Отчёт по домашнему заданию №2

## a. Общая идея решения

Приложение моделирует систему **импорта и экспорта банковских данных** (счета, операции, категории) в различных форматах — **CSV, JSON, YAML**. 
Основная цель — реализовать архитектуру, поддерживающую расширяемость, низкую связанность и повторное использование кода.

### Основной функционал:
- Импорт данных о счетах, категориях и операциях из внешних файлов. 
- Экспорт этих данных в выбранный формат. 
- Подсчёт метрик (через `MetricsCollector`). 
- Хранение данных в памяти (через `InMemoryRepository` реализации). 
- Инверсия зависимостей с помощью собственного DI-контейнера (`Container`).

Фасад `ImportFacade` объединяет логику импорта из разных источников (CSV, JSON, YAML), предоставляя единый интерфейс для остальной системы. 
Фабрика `DomainFactory` инкапсулирует создание доменных объектов, а пакет `export` реализует паттерн **Visitor** для расширяемости форматов экспорта.

## b. Принципы **SOLID** и **GRASP**

| Принцип | Где реализован | Как реализован |
|----------|----------------|----------------|
| **S — Single Responsibility** | `CsvImporter`, `YamlImporter`, `JsonImporter`, `DomainFactory` | Каждый класс отвечает за одну задачу — импорт, создание объектов или экспорт. |
| **O — Open/Closed** | `ExportVisitor` и его наследники | Добавление нового формата экспорта требует создать новый Visitor, не изменяя существующие классы. |
| **L — Liskov Substitution** | Все классы, реализующие `Importer` и `ExportVisitor` | Любой конкретный импортёр или экспортёр может подставляться вместо интерфейса. |
| **I — Interface Segregation** | Интерфейсы `Importer`, `ExportVisitor` | Разделение контрактов на мелкие, независимые интерфейсы. |
| **D — Dependency Inversion** | `Container` и использование абстракций `Importer` / `Repository` | Классы зависят от интерфейсов, а не от конкретных реализаций. Зависимости внедряются через DI. |

### GRASP-принципы:
| Принцип | Где проявлен | Обоснование |
|----------|---------------|-------------|
| **Creator** | `DomainFactory` | Класс создаёт объекты, для которых имеет нужные данные и контекст. |
| **Controller** | `ImportFacade` | Управляет потоком данных между слоями, делегируя конкретные задачи. |
| **Low Coupling / High Cohesion** | Весь проект | Каждая подсистема (import, export, domain, repository) слабо связана и решает свои задачи. |
| **Polymorphism** | `Exporter`, `Importer` | Разные реализации работают через общий интерфейс. |
| **Pure Fabrication** | `Container`, `MetricsCollector` | Введены искусственно для снижения связанности и упрощения архитектуры. |

## c. Использованные паттерны **GoF**

| Паттерн | Где реализован | Важность и роль |
|----------|----------------|-----------------|
| **Factory Method / Abstract Factory** | `DomainFactory` | Централизует создание доменных объектов (аккаунтов, категорий, операций), упрощает добавление новых типов. |
| **Strategy** | `Importer` + реализации (`CsvImporter`, `JsonImporter`, `YamlImporter`) | Позволяет выбирать стратегию импорта во время выполнения. |
| **Visitor** | `ExportVisitor` и наследники (`JsonExportVisitor`, `CsvExportVisitor`, `YamlExportVisitor`) | Обеспечивает расширяемость форматов экспорта без изменения доменных классов. |
| **Facade** | `ImportFacade` | Объединяет логику импорта в единую точку доступа, упрощая использование системы. |
| **Singleton** *(частично)* | `Container` | DI-контейнер может существовать в единственном экземпляре в приложении. |
| **Dependency Injection (DI)** | `Container` | Снижает связанность между компонентами, облегчает тестирование и замену реализаций. |

## d. Структура проекта

```
src/com/hsebank/
 ├── Main.java                  — точка входа
 ├── importer/                  — импорт данных
 │    ├── ImportFacade.java
 │    ├── Importer.java
 │    ├── CsvImporter.java
 │    ├── JsonImporter.java
 │    └── YamlImporter.java
 ├── export/                    — экспорт данных (Visitor)
 │    ├── ExportVisitor.java
 │    ├── CsvExportVisitor.java
 │    ├── JsonExportVisitor.java
 │    └── YamlExportVisitor.java
 ├── factory/                   — фабрика доменных объектов
 │    └── DomainFactory.java
 ├── di/                        — Dependency Injection контейнер
 │    └── Container.java
 ├── repository/                — репозитории в памяти
 │    ├── InMemoryBankAccountRepository.java
 │    ├── InMemoryCategoryRepository.java
 │    └── InMemoryOperationRepository.java
 ├── domain/                    — доменные классы (Category, Operation, Account)
 └── metrics/                   — сбор метрик (MetricsCollector)
```

## 📄 Итоги

- Архитектура следует принципам **SOLID** и **GRASP**. 
- Использованы ключевые **паттерны GoF**: *Factory*, *Strategy*, *Visitor*, *Facade*, *DI*. 
- Код легко расширяем и поддерживаем — можно добавить новые форматы, сущности или источники данных без изменения существующих модулей. 
- Реализация демонстрирует понимание модульности, инверсии зависимостей и принципов проектирования.
